全局过滤器

在实际项目中，有一类需求很常见，例如

  • 软删除

    业务层面的删除操作，并非对应于数据库的删除操作，而是为数据设置一个“已删除”的标记，绝大部分查询操作不查询有此状态的数据。这样，被标记为“已删除”的数据将会被屏蔽，直到“已删除”的状态被撤销为止。

  • 多租户

    数据具备一个“所属租户”字段，如果系统的登陆信息也可以解析出“当前租户”，则利用“当前租户”过滤查询数据，从而让不同租户的访问者登陆后看到不同的数据。

  • 数据授权

    数据可能通过某些和授权相关的关联，间接地和用户有关系。利用登陆身份，可以做到让每个用户仅能查询他被授权可以访问的数据。

以上这类需求，本质上是一样的，这是一种隐式地为查询植入where条件的诉求，Jimmer提供全局过滤器解决这个问题。

提示

如果读者熟悉Oracle数据库，可以用VPD来类比理解。Jimmer全局过滤器为所有数据库引入了类似于Oracle VPD的能力。

以上这类需求中，多租户复杂度比较适中，本文以多租户为例，讲解全局过滤器。

备注

在某些多租户系统中，不同租户的数据需要分表甚至分库存储。

如果有这方面的诉求，请结合sharding-jdbc使用。在JDBC层面而非ORM层面解决分库分表的问题。

提供过滤器​

提供抽象实体基类​

首先，提供一个MappedSuperclass超类，让所有需要多租户管理的实体类都继承它

  • Java
  • Kotlin
@MappedSuperclass
public interface TenantAware {

    String tenant();
}
@MappedSuperclass
interface TenantAware {

    val tenant: String
}

定义过滤器​

下面的示例中，TenantProvider类的Java方法get()和kotlin属性tenant用于返回当前操作着所属租户。

  • Java下，拦截器需实现org.babyfish.jimmer.sql.filter.Filter接口，
  • Kotlin下，拦截器需实现org.babyfish.jimmer.sql.kt.filter.KFilter接口。

如果使用Spring托管，代码方式如下：

  • Java
  • Kotlin
@Component
public class TenantFilter implements Filter<TenantAwareProps> {

    protected final TenantProvider tenantProvider;

    public TenantFilter(TenantProvider tenantProvider) {
        this.tenantProvider = tenantProvider;
    }

    @Override
    public void filter(FilterArgs<TenantAwareProps> args) {
        String tenant = tenantProvider.get();
        if (tenant != null) {
            args.where(args.getTable().tenant().eq(tenant));
        }
    }
}
@Component
class TenantFilter(
    protected val tenantProvider: TenantProvider
) : KFilter<TenantAware> {

    override fun filter(args: KFilterArgs<TenantAware>) {
        tenantProvider.tenant?.let {
            args.apply {
                where(table.tenant.eq(it))
            }
        }
    }
}

TenantFilter首先从提取当前操作者的所属租户，如果所属租户非null，则使用它来过滤数据，只查询和指定租户匹配的数据。

配置过滤器​

将过滤器挂接到SqlClient对象上，即可生效

  • Java
  • Kotlin
@Bean
public JSqlClient sqlClient(
    List<Filter<?>> filters,
    ...省略其他参数...
) {
    return JSqlBuilder
        .newBuilder()
        .addFilters(filters)
        ...省略其他配置...
        .build();
}
@Bean
fun sqlClient(
    filters: List<KFilter<?>>,
    ...省略其他参数...
): KSqlClient =
    newKSqlClient {
        addFilters(filters)
        ...省略其他配置...
    }

虽然这里仅示范了一个Filter/KFilter，但是实际项目中可能有很多个过滤器。

所以，这里使用集合，让Spring注入所有的Filter/KFilter。

过滤聚合根对象​

过滤聚合根对象是全局过滤器最简单的用法。

假设Book实体继承了TenantAware，用法如下

  • Java
  • Kotlin
List<Book> books = sqlClient.getEntities.findAll(Book.class);
val books = sqlClient.entities.findAll(Book::class);
或
  • Java
  • Kotlin
BookTable book = BookTable.$;
List<Book> books = sqlClient
    .createQuery(book)
    .select(book)
    .execute();
val books = SqlClient
    .createQuery(Book::class) {
        select(table)
    }
    .execute()

生成的SQL如下

假设Book实体继承了TenantAware，用法如下

这会导致如下两句SQL被生成

    select 
        tb_1_.ID, tb_1_.FIRST_NAME, tb_1_.LAST_NAME, tb_1_.GENDER

  2. 查询关联对象

不难发现，这里使用了对象抓取器。虽然对象抓取器没有对关联属性Author.books施加任何字段级过滤器，但Book类的全局过滤器仍然生效。导致，最终生产的SQL仍然过滤了租户信息。

信息

这里通过对象抓取器演示了关联对象的过滤。事实上，Jimmer提供的GraphQL功能同样可以利用全局过滤器过滤关联对象，二者效果一样，不再赘述。

启用禁用过滤器​

这节我们以软删除为例，假设有两个过滤器类

  • UndeletedFilter: 查询状态没有被设置成“已删除”的数据
  • DeletedFilter: 查询状态已经被设置成“已删除”的数据

软删除相关的过滤器非常简单，无需展示这两个过滤器的代码。

现在，我们把这两个过滤器挂接到SqlClient上

  • Java
  • Kotlin
@Bean
public JSqlClient sqlClient(
    List<Filter<?>> filters,
    ...省略其他参数...
) {
    return JSqlBuilder
        .newBuilder()
        .addFilters(new UndeletedFilter())
        .addDisabledFilters(new DeletedFilter())
        ...省略其他配置...
        .build();
}
@Bean
fun sqlClient(
    filters: List<KFilter<?>>,
    ...省略其他参数...
): KSqlClient =
    newKSqlClient {
        .addFilters(UndeletedFilter())
        .addDisabledFilters(DeletedFilter())
        ...省略其他配置...
    }

显然，如果两个过滤器同时使用的话，将查询不到任何数据。

所以，DeletedFilter虽然被添加上去了，但是并没有启用。而UndeletedFilter是启用状态，默认情况下，所有业务关心未被删除的数据。

假如有少量业务与众不同，不查看未被删除的数据，专门查看已经被删除的数据

  • Java
  • Kotlin
JSqlClient tmpSqlClient =
    sqlClient.filters(it -> {
        it
            .disableByTypes(UndeletedFilter.class);
            .enableByTypes(DeletedFilter.class);
    });
val tmpSqlClient = 
    sqlClient.filters {
        disableByTypes(UndeletedFilter::class)
        enableByTypes(DeletedFilter::class)
    }

在这里，我们把原本启用的UndeletedFilter禁用了，而把原本禁用的DeletedFilter启用了，创建了全新的tmpSqlClient。

tmpSqlClient的行为和默认的sqlClient刚好相反，不查询未被删除的数据，专门查询已经被删除的数据。

Sharding过滤器​

默认情况下，使用简单查询API按照id查询对象不会应用过滤器，这里既包括用户的sqlClient.entities.findById(s)行为，也包括jimmer内部利用对象缓存把id转化为对象的行为。

危险

注意:

这里指直接按照id查询对象，而不是对象抓取器器针对基于外键的多对一关联的查询。

即使外键数据非null, 如果父对象不满足其过滤条件，对象抓取器也是会认为父对象为null。

此时，如果多对一关联属性不允许为null，会导致异常。

查询基于外键的多对一关联有两种方法：

  1. 使用对象抓取器，显式抓取某个基于外键的多对一关联，或隐式地使用对象抓取器的allTableFields()方法

  2. 不使用对象抓取器，而是直接查询表对象。在这种情况下，所有的外键都会被设置为只有id的父对象或null

其中，第一种情况下会使用过滤器，第二种情况不会使用过滤器。

大部分情况下，按照id查询对象忽略过滤器是正确的，因为id唯一确定了对象。

然而，如果JDBC层面使用了sharding-jdbc技术，且过滤器适用的字段就是sharding-jdbc中分库分表的字段，这时，仅仅根据id查询对sharding-jdbc而言是灾难。

为了解决这个问题，让过滤器实现ShardingFilter接口(Java)或KShardingFilter接口(Kotlin)即可，该接口无任何行为，仅仅做类型标记。

一旦过滤器是ShardingFilter或KShardingFilter，简单查询API所对应的基于id的简单查询，无论是用户的sqlClient.entities.findById(s)行为还是Jimmer内部行为，将不再忽略过滤此过滤器，坚持应用过滤条件。

这可以保证最终SQL一定包含sharding-jdbc所需的sharding字段，从而实现只查询单个分库，而非所有分库都查询一次。

过滤器和缓存​

前文提到，全局过滤器可以和对象抓取器或GraphQL配合使用，用于过滤关联对象。

这表示，不同的操作者，可以看到不同的对象关联。

然而，在外部缓存一文中，我们提到Jimmer支持关联缓存看起来是这样的

keyvalue
Author.boooks-1[1, 2, 3]
Author.boooks-2[4, 5, 6]

这表示，和id为1的作者相关的书籍是1、2和3，而和id为2的作者相关的书籍是4、5和6。

这个例子中，Author.books这个多对多关联被缓存所描述，然而，所有操作者看到的缓存数据是一样的，这和全局过滤器可以让不同的操作者看到不同的对象关联不相符。

为了，Jimmer支持更高级的缓存功能，和全局过滤器配合，让不同的操作者看到不同的缓存。这样的缓存看起来应该如此

keyhashvalue
Author.boooks-1{}[1, 2, 3]
{"tenant":"a"}[1, 3]
{"tenant":"b"}[2]
Author.boooks-2{}[4, 5, 6]
{"tenant":"a"}[4, 6]
{"tenant":"b"}[5]

在这里，我们看到了缓存条目的膨胀，如果活跃的过滤器参数越多，消耗的缓存就越多。

所以，仅应针对重要的关联才使用这种缓存能力，且推荐使用redis集群。

过滤器是否支持缓存缓存是否支持参数化最终效果
否NA不使用用关联缓存
是否使用关联缓存，但仅缓存无参过滤器的结果
是是使用关联缓存，且缓存所有过滤器的结果 (就是上面介绍的这种缓存条目膨胀后的缓存)

由此可见，如果要为所有过滤器参数提供缓存，开发人员需要

  • 让过滤器支持缓存
  • 让缓存支持参数化

让过滤器支持缓存​

为了使过滤器支持缓存

  • Java下，需实现org.babyfish.jimmer.sql.filter.CacheableFilter接口，
  • Kotlin下，需实现org.babyfish.jimmer.sql.kt.filter.KCacheableFilter接口。
  • Java
  • Kotlin
@Component
public class TenantFilter implements Filter<TenantAwareProps> {

    protected final TenantProvider tenantProvider;

    public TenantFilterForNonCacheMode(TenantProvider tenantProvider) {
        this.tenantProvider = tenantProvider;
    }

    @Override
    public void filter(FilterArgs<TenantAwareProps> args) {
        String tenant = tenantProvider.get();
        if (tenant != null) {
            args.where(args.getTable().tenant().eq(tenant));
        }
    }

    @Override
    public SortedMap<String, Object> getParameters() {
        String tenant = tenantProvider.get();
        if (tenant == null) {
            return null;
        }
        SortedMap<String, Object> map = new TreeMap<>();
        map.put("tenant", tenant);
        return map;
    }

    ...忽略其他方法...
}
@Component
class TenantFilter(
    protected val tenantProvider: TenantProvider
) : KFilter<TenantAware> {

    override fun filter(args: KFilterArgs<TenantAware>) {
        tenantProvider.tenant?.let {
            args.apply {
                where(table.tenant.eq(it))
            }
        }
    }

    override fun getParameters(): SortedMap<String, Any>? =
        tenantProvider.tenant?.let {
            sortedMapOf("tenant" to it)
        }

    ...忽略其他方法...
}

为CacheableFilter/KCacheableFilter实现getParameters()方法，返回一个map。

在本文的例子中，仅使用了一个过滤器。如果某个实体类被多个过滤器过滤，系统会自动把多个过滤器合并成一个过滤器。

  • 任何一个过滤器不支持缓存，最终合并后的过滤器不支持缓存
  • 如果合并后的过滤器仍然支持缓存，那么合并后的过滤器的getParameters()方法返回的map是所有原始过滤器的getParameters()方法返回的map的并集

如果关联缓存支持参数化的，也就是说缓存是一个嵌套的双层KeyValue结构，那么合并后的过滤器返回map，被序列化为字符串后，即可作为内层KeyValue结构的键。

让缓存支持参数化​

为让缓存是嵌套的双层KeyValue结构，缓存需要实现Cache.Parameterized<K, V>接口

Cache.java
// 普通缓存
public interface Cache<K, V> {

    ...省略方法...

    // 参数化的缓存
    interface Parameterized<K, V> extends Cache<K, V> {

        ...省略方法...
    }
}

创建缓存的代码如下

  • Java
  • Kotlin
Cache<K, V> cache = new ChainCacheBuilder<K, V>()
    .add(level1Binder)
    .add(level2Binder)
    ... 
    .add(levelNBinder)
    .build()
Cache<K, V> cache = ChainCacheBuilder<K, V>()
    .add(level1Binder)
    .add(level2Binder)
    ... 
    .add(levelNBinder)
    .build()

即使使用参数化的缓存，也只需要声明Cache<K, V>类型的变量。开发人员只需要创建Cache.Parameterized<K, V>类型的对象实例即可。

为了构建多级缓存，需构建Binder对象

  • 如果缓存能自动loading，如caffine, guava

    普通缓存的Binder参数化缓存的Binder
    Java需要实现的接口LoadingBinder<K, V>LoadingBinder.Parameterized<K, V>
    Kotlin需要实现的接口KLoadingBinder<K, V>KLoadingBinder.Parameterized<K, V>

  • 如果缓存不能自动loading，需要jimmer调用其set, 如redis

    普通缓存的Binder参数化缓存的Binder
    Java需要实现的接口SimpleBinder<K, V>SimpleBinder.Parameterized<K, V>
    Kotlin需要实现的接口KSimpleBinder<K, V>KSimpleBinder.Parameterized<K, V>
    Java推荐的抽象基类AbstractRemoteValueBinder<K, V>AbstractRemoteHashBinder<K, V>
    kotlin推荐的抽象基类AbstractKRemoteValueBinder<K, V>AbstractKRemoteHashBinder<K, V>
危险

构建多级缓存时

  • 如果所有Binder都是参数化的，最终创建的缓存也是参数化的
  • 如果所有Binder都是非参数化的，最终创建的缓存也是非参数化的
  • 如果两种Binder混用，报错

为AbstractRemoteHashBinder或AbstractKRemoteHashBinder提供派生类，可以快速实现参数化的缓存Binder，进一步实现参数化的缓存。

这里不再赘述，请参考项目附带例子中的RedisHashBinder实现，请点击Java示范链接或Kotlin示范链接

最后，借助于RedisHashBinder，我们可以把关联缓存和计算属性缓存配置为参数化缓存

  • Java
  • Kotlin
@Bean
public JSqlClient sqlClient(
    RedisTemplate<stirng, byte[]> redisTemplate,
    ObjectMapper objectMapper,
    ...其他参数...
) {
    return JSqlBuilder
        .newBuilder()
        .setEntityManager( // 设置Cache前，需先设置EntityManager
            JimmerModule.ENTITY_MANAGER // 自动生成的常量
        )
        .setCaches(it -> {
            it.setCacheFactory(
                new CacheFactory() {

                    // 对象缓存，普通缓存即可
                    @Override
                    public Cache<?, ?> createObjectCache(ImmutableType type) {
                        return new ChainCacheBuilder<>()
                                .add(new CaffeineBinder<>(512, Duration.ofSeconds(1)))
                                .add(
                                    new RedisValueBinder<>(
                                        redisTemplate, 
                                        objectMapper, 
                                        type, 
                                        Duration.ofMinutes(10)
                                    )
                                )
                                .build();
                    }

                                        objectMapper, 
                                        prop, 
                                        Duration.ofMinutes(5)
                                    )
                                )
                                .build();
                    }

参数化缓存一致性​

参数化关联缓存缓存的结果，不仅会受到表之间关系的影响，还会受到关联对象过滤器影响。

对于普通过滤器而言，你可以施加任何SQL过滤规则。

然而，支持缓存的过滤器不同，被施加了限制，既不支持表连接，也不支持子查询。

这种限制意味着，支持缓存的过滤器只能使用被过滤表的字段实施筛选。反过来讲，除表之间的关系外，只有被过滤表的数据变更，才会影响当前过滤器所对应的关联缓存。

CacheableFilter/KCacheableFilter还需实现一个isAffectedBy方法，用于询问被过滤表的数据变更是否会对当前过滤器有影响。

  • Java
  • Kotlin
@Component
public class TenantFilter implements Filter<TenantAwareProps> {

    @Override
    public boolean isAffectedBy(EntityEvent<?> e) {
        // 如果被过滤表的tenant字段被修改，则当前过滤器受影响
        return e.getChangedFieldRef(TenantAwareProps.TENANT) != null;
    }

    ...忽略其他方法...
}

过滤器和计算属性​

在计算属性及缓存一文中，我们讨论过如何实现计算属性和相关缓存。

TransientResolver或KTransientResolver接口支持一个方法getParameterMapRef()，返回类型Ref<SortedMap<String, Object>>。

此方法具备默认实现，并非必须覆盖。默认行为返回一个非null的Ref，但内部的值为null。

getParameterMapRef方法返回的Ref缓存是否支持参数化
nullNA不使用计算缓存
Ref本身非null，但内部的map为null或长度为0NA使用计算缓存，缓存当前无参过滤器对应的计算结果
Ref内部的map长度非0否不使用计算缓存
Ref内部的map长度非0是使用计算缓存，缓存当前有参过滤器对应的计算结果

对于最后两种情况，计算缓存看起来可能是这样的
